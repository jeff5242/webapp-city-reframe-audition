## 二、詳細規則 spec（示範 3 條，展示抽取深度）

### DOC-A14 — 附錄十四「建材設備等級表」必附
```yaml
rule_id: DOC-A14
category: 文件完整性
checkability: 可機械檢查
法源: 111年版 附錄清單 第十四項（備註：必附）
原文依據: "| 十四 | 建築工程建材設備等級表 | 必附 |"
檢查條件: 文件目錄/附錄中存在「建材設備等級表」且非空
期望值: 存在
資料來源: OCR 目錄頁 + 附錄章節標題比對
嚴重度: 必檢（缺件不通過）
檢查邏輯: |
  if not find_section(附錄, ["建材設備等級表","建材設備"]):
      fail("附錄十四 建材設備等級表 未檢附（111版必附）")
信心: 高
備註: 附錄清單為固定 24 項，"必附/若無則免附/請擇一" 語意明確，抽取穩定
```

### FMT-04 — 面積/長度單位規範
```yaml
rule_id: FMT-04
category: 表單格式
checkability: 可機械檢查
法源: 111年版 計畫書製作規範 · 數字格式
原文依據: "單位：面積用 m²；公尺用 M"
檢查條件: 面積數值後單位為 m²（非 平方公尺/㎡ 混用）；長度單位為 M
期望值: 面積→m²、長度→M
資料來源: 正規表達式掃描數值+單位樣態
嚴重度: 建議（格式偏差，非實質違規）
檢查邏輯: |
  for token in 面積欄位:
      if unit not in ["m²"]: warn(f"面積單位應為 m²，發現 {unit}")
信心: 高
備註: 屬「格式一致性」，可自動；不影響審議實質，列建議級
```

### REW-01 — 容積獎勵上限（半自動範例）
```yaml
rule_id: REW-01
category: 數值計算
checkability: 半自動（AI 抽數值 + 規則驗上限；個案合理性需人審）
法源: 111年版 修訂重點·6「依都更條例第65條修正獎勵上限」
原文依據: "申請容積獎勵項目及額度 — 依都更條例第65條修正獎勵上限"
檢查條件: 合計容積獎勵 ≤ 該案基準容積 × 法定上限比率（依區位/案型）
期望值: 不逾上限
資料來源: 審議資料表(基準容積、合計獎勵樓地板) + 容積獎勵檢核表
嚴重度: 必檢
檢查邏輯: |
  上限 = 基準容積 * 法定上限比率(依都更條例§65 + 自治條例)
  if 合計獎勵樓地板 > 上限: fail("容積獎勵逾上限")
  else if 逼近上限(>95%): flag("接近上限，建議委員複核各獎勵項目證明")
信心: 中
備註: ⚠ 上限比率隨區位/案型/年版不同，需接 version_selector；各獎勵「項目是否符合」仍需人審 → 標半自動
```

## 三、方法與規模化（怎麼「大量且正確」）

**產線（agentic）**：
```
條文切條 → [分類 agent] 可檢查?類型? → [抽取 agent] 條文→結構化 spec(條件/門檻/法源)
  → [產生] 檢查邏輯(playbook config) → [critic agent] 對 gold case 驗證這條有無誤解條文
  → 人工簽核 → 進 playbook（逐年版 107/108/111/113）
```

**correctness 的守門(不能省)**：
1. 每條規則**強制附法源原文**（可回溯，杜絕 AI 編造）。
2. 分類先行——**只把「可機械檢查」的自動化**，裁量條款一律退回人審，不硬做。
3. 生成規則需通過 **gold case 回歸測試**才能上線（沿用你 eval harness）。
4. 半自動規則（如 REW-01）明確標「AI 抽數值、人審合理性」。

**真正的長期價值**：接 `regulation-versions.md` + RAG 法規庫 → **條文改版自動 diff → 標出哪些規則要跟改**，維護才可規模化。

## 四、誠實的限制
- 本 PoC 的 16 條，🟢 可直接自動化的約 14 條、🟡 半自動 2 條；裁量條款不產規則。
- 「必附/門檻/格式/遮蔽」這類語意明確的抽取信心高；「合理性/妥適性」類無法機械化。
- 生成規則**一律草稿**，須承辦複核；建議先擴充「文件完整性 + 格式 + 個資」這三類（低風險、高覆蓋）。
