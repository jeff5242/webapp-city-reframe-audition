---

## 一、一頁摘要（給主管的三句話）

1. **核心引擎已上線可用**：14 條法規規則引擎（Track A）+ AI 語意審查（Track B）皆已接入正式流程並顯示於報告；OCR 本週完成品質與速度大改造。
2. **我們目前的層級**：能做到「**規則檢查 + AI 語意標記**」，等同業界的「自動初審助理」。但相較成熟合規審查系統，我們**缺三塊差異化能力**：證據引用（把每個問題連回原文位置）、信心分數分流、人工複核佇列。
3. **下一階段目標**：從「自動標記問題」升級為「**可舉證、可信賴、可人工協作**的審議系統」——這三塊正是讓審議委員敢採用、敢背書的關鍵。

## 三、能力缺口分析（對照業界成熟做法）

調查了 GitHub 上 30+ 個 PDF 合規/審查/文件 AI 專案後，歸納出「業界成熟做法有、我們目前沒有」的關鍵缺口：

### 🔴 缺口 1：證據引用（Evidence Grounding）— 最高優先

**現況**：AI 找到問題後，只給一段文字描述，**無法點回原文精確位置**。審議委員無從快速查證。

**業界做法**：
- **Anthropic Citations API**（我們已是 Anthropic 用戶）：把 PDF 當 document 傳入，Claude 回傳**保證不 hallucinate** 的原文引用 + 頁碼。
- **OpenContracts (PAWLS)**：每個問題錨定到「頁碼 + 正規化座標 + 原文 span」。

**價值**：讓紅框標註連回 AI 發現（目前紅框只標規則命中），每個 finding 可一鍵跳到原文舉證。**這是讓委員敢採信的前提**。

### 🔴 缺口 2：信心分數與分流（Confidence + HITL）

**現況**：所有 AI 發現同等對待，無「這條我很確定 / 這條需人工確認」的區分。

**業界做法**：
- 複合信心分數 = 規則結果 × LLM 結果（兩者一致→高信心；分歧→強制人工）。
- **時效性子分數**：從我們既有的 `version_selector`（報核日期→年版）導出，是規則審查工具的**差異化亮點**。
- 依「信心 × 嚴重度」路由：高信心低風險自動核可，低信心或 PII/高風險強制人工。

### 🟡 缺口 3：規則外部化（Policy-as-Config）

**現況**：各年版法規門檻寫在程式碼裡（`field_auditor` 呼叫時未傳年版，實質固定跑 111 年）。

**業界做法**：規則抽成**每年一份 playbook 設定檔**，法規修訂時改設定不改程式。同時採用三級旗標（GREEN 合規 / YELLOW 輕微偏差 / RED 必改違規）比現有 pass/fail/warn 更貼合審議語境。

### 🟡 缺口 4：評測基準（Evaluation Harness）

**現況**：**零評測**。無法量化「AI 審查的準確率有多高」。

**業界做法**：建立小型 gold set（人工標註的標準答案），用 DeepEval / promptfoo 量測 precision/recall，進 CI。**對主管報告時，這是回答「這東西到底準不準」的唯一方式**。

### 🟢 缺口 5：OCR 後校正 + RAG 法規知識庫（進階）

- **OCR 後校正**：對低信心頁面用多模態 LLM（圖+文）修正錯字、重建表格（KAP 繁中專用做法）。⚠️ 注意：OCR 已不錯時純文字校正會反效果，須信心門檻 gated。
- **法規即知識庫**：用 LlamaIndex 把法規條文建索引，每個 finding 檢索並**引用對應法條**，配 Judge agent 驗證年版與 grounding。
