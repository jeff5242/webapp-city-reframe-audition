# 開發內容說明｜都更報核協審 AI Agent 服務（第一期）

> **用途**：補助計畫書「貳、計畫內容與實施方法 — 三、實施方法」之詳版，供評審理解**開發什麼、與都更審議題目如何關聯、預計怎麼研發**。可整併回 `補助計畫書_都更AI協審_第一期_v0.2.md`。
> **技術基礎**：本計畫在既有運行系統上開發（Python/FastAPI，雙軌審查架構）。技術數據為本專案真實實測。

## 〇、整體技術架構與研發總策略

**系統定位**：以「**都更處 Orchestration Agent**」為核心的代理式協審服務。承辦上傳事業計畫書／權利變換計畫書 PDF，Agent 透過**任務分解（Task Decomposition）**編排一連串工具呼叫（讀表 → 格式校正 → 法規核算 → 查詢法規 → 產生協審報表），最終交承辦覆核。

**雙軌並行、AI 與確定性互補**（核心設計理念）：

| 軌道 | 內容 | 特性 | 對應分項 |
|---|---|---|---|
| **A 軌（AI 視覺理解）** | 地端 VLM 讀圖直出審議表結構化欄位 | 讀得懂掃描影像/密集表格，但有準確率上限 | A 分項 |
| **B 軌（確定性引擎）** | 格式母版校正 + 14 條法規規則 + RAG | 100% 可解釋、可稽核、不受模型準確率影響 | B 分項 |

> **關鍵**：A 軌讀出的數值交由 B 軌做上下文驗證（超法定上限、日期矛盾、數值離譜 → 回饋重讀）。**B 軌對 A 軌的錯誤兜底**——這是「政府審查工具容不得 AI 幻覺」的解法，也是本案相較純 AITool 的護城河。

**研發總策略（三大原則）**：
1. **既有基礎躍升，非從零研究**：既有系統已具規則引擎、審議表判讀、格式校正 PoC，本計畫聚焦「準確率提升 ＋ Agent 化 ＋ 平台整合 ＋ Trust AI」。
2. **確定性優先、AI 輔助**：能用規則/母版確定判定的（格式、門檻、版本）絕不交給 AI；AI 只做「讀圖」與「語意協審初判」，且一律人工覆核。
3. **主權地端、資料不出境**：VLM 推理與微調於台灣境內地端執行；真實案例個資（地號/姓名）去識別化後才進語意層。

## 二、A 分項｜審議表地端 VLM 辨識微調 ＋ 協審 Agent

**與題目之關聯**：審議資料表是審議的核心單據——一張欄位密集的表單（案名、地號、面積、容積、獎勵、停車位、日期、實施者等 20 餘欄），承辦過去須逐欄以肉眼判讀。此分項讓 AI「讀懂這張表」，是整個協審流程的入口。

### A1　gold 標註集與弱監督自動回標管線

- **與題目之關聯**：模型要學會讀審議表，需要「圖 → 正確欄位」的標註資料（gold）。都更審議表版面固定、且平台已有結構化欄位，正好讓標註可自動化。
- **技術內容**：
  - 建立審議表影像（300 DPI PNG）→ 欄位 JSON 之標註 schema（20 餘欄，含空值處理、全半形/千分位規範）。
  - **弱監督自動回標**（本案關鍵成本優勢）：
    - *官方範本母版對位*：以本計畫產出之**機讀格式母版**（113 年版，事業計畫書 18 章 24 附錄）對位審議表已知欄位區域，程式自動框定「欄位 → 值格」。
    - *平台結構化欄位回標*：審議服務平台案件詳情本身含結構化欄位（獎勵、面積、停車、費用、日期），以此為 ground truth 自動與 PDF 對齊回標，**由腳本產生標籤、非人工逐筆**。
    - *少量人工校驗*：工程師抽樣校驗邊界案例。
- **預計研發方式**：先以既有十餘筆 gold 建立管線與基準，再以自動回標將 gold 擴至上百筆；資料版本以 DVC/git 管理；標註品質以 inter-annotator 抽驗與規則檢核（如面積>0、日期合法）把關。
- **交付/查核**：gold 標註集（上百筆）＋自動回標腳本＋標註規範（查核點 A1，Q2）。

### A2　地端 VLM 部署與微調

- **與題目之關聯**：把零樣本「讀得半對」的 VLM，針對都更審議表微調到「讀得準」，是準確率 51%→85%+ 的主力槓桿。
- **技術內容**：
  - 基座採**可離線、可稽核之開源權重視覺語言模型**（非中資來源），地端部署（單至雙卡 H100/A100 等級，OpenAI 相容推論服務）。
  - **關鍵參數已實測**：輸入解析度 `image_max_pixels` 對密集表格辨識率影響巨大——實測 8M px 為甜蜜點（過低壓糊、過高反劣）；production 推論須設對此參數。
  - 微調採 **LoRA/QLoRA**（LLaMA-Factory sharegpt 多模態格式），資料為 A1 之 gold；以「代理式工作流程反覆迭代」逐步收斂。
- **預計研發方式**：先跑零樣本建立基準（省算力），再 LoRA 微調；以 gold 測試集逐輪評測逐欄準確率；難欄位以資料增補與 prompt 工程補強；**模型來源橫向實測作為選型依據**（已完成初步：中資來源繁中較佳但受政策限制，非中資來源繁中落差且有幻覺 → 以在地標註微調補足繁中正確性）。
- **交付/查核**：微調後模型 ＋ 逐欄準確率 ≥【85%】評估報告（查核點 A2，Q4）。

### A3　難點欄位補強（跨子表推理）

- **與題目之關聯**：**報核日期**是決定「適用哪一年版法規」的關鍵，但它藏在「辦理過程」子表的特定列，需跨區塊推理——這是純逐欄辨識最容易錯、卻對審議最關鍵的欄位。
- **技術內容**：
  - 以「VLM 讀出整張辦理過程子表 ＋ 程式規則判定報核列」的架構打通（語意完整區塊裁切，而非全頁盲切——實測全頁盲切反而傷準確率）。
  - 報核日期優先序判定：①審議資料表「辦理過程」最新報核列 → ②申請書/切結書報告日 → ③填表日期 → ④檔名日期。
- **預計研發方式**：對難點欄位做**精準語意裁切**（只裁該子表），VLM 讀小塊乾淨圖再以規則定位目標列；以規則引擎交叉驗證年版一致性。
- **交付/查核**：難點欄位模組 ＋ 報核日期判定準確率報告。

### A4　Orchestration Agent 協審工作流程

- **與題目之關聯**：把「讀表、格式檢查、法規核算、查法規、產報表」這一串承辦要做的事，交給一個會**規劃任務、呼叫工具、自我評判**的 Agent 編排——這是本案身為「AI Agent 整合服務」的核心。
- **技術內容**：
  - Orchestration Agent 以 **Task Decomposition** 規劃協審任務清單；透過 **tool-use** 呼叫工具：`讀審議表(VLM)`、`格式校正(母版)`、`法規核算(規則引擎)`、`查法規(RAG)`、`產生協審報表`。
  - 具**評判與迭代**：工具回報異常（超上限/日期矛盾）→ Agent 觸發回饋重讀或標「待覆核」。
  - 記憶（案件上下文）與審計軌跡（每步輸入/輸出/依據留存）。
- **預計研發方式**：以既有 Track B（llm_auditor/field_auditor/cross_doc_comparator）為工具基礎，包裝為 Agent 可呼叫之標準工具服務（tool API）；先固定流程（deterministic pipeline）跑通，再開放 Agent 動態規劃；以真實案例回歸測試確保流程穩定。
- **交付/查核**：Orchestration Agent 協審流程上線（查核點 A4，Q3）。

## 三、B 分項｜格式校正 ＋ 規則引擎 ＋ LLM Wiki RAG 協審

**與題目之關聯**：審議不只看「表填什麼」，更要看「文件合不合規、數字對不對法規、用的是不是正確年版」。這分項是**確定性審查**，不靠 AI 準確率，直接命中承辦的高頻退件理由。

### B1　官方範本機讀格式母版與校正引擎

- **與題目之關聯**：都更計畫書須照都更處官方範本（18 章 24 附錄），缺章節/缺附錄/擇一未填是常見退件原因。此項把「格式合規」變成確定性檢查。
- **技術內容**（**本計畫已完成 PoC**）：
  - 解析官方 113 年版範本 ODT → **機讀格式母版**（章節/附錄清單＋必附性，由標題「（請擇一填寫）/（若無則免附）」自動判定）。已產出事業計畫書（18 章 24 附錄）、權利變換、事業概要三份 schema。
  - 校正引擎比對上傳文件結構 → 缺必附章節（FMT-CH）、缺必附附錄（FMT-AP）、擇一群未擇一（FMT-XOR）。
  - **偵測與檢核分離**：偵測不到時不判定，絕不誤報整份缺件。
- **預計研發方式**：母版由官方 ODT 一鍵重建（範本改版即重跑）；校正結果併入協審報表；逐年版/各縣市可複製（換 ODT 即可）。
- **交付/查核**：格式校正引擎，官方必附章節/附錄涵蓋 100%（查核點 B1，Q2）。

### B2　法規規則外部化與數值核算

- **與題目之關聯**：容積獎勵上限、同意比例、停車位計算、日期一致性等，是審議的量化門檻；規則零散且逐年版不同。
- **技術內容**：
  - 既有 **14 條規則引擎**（文件完整性、表單、PII、一致性、計算）＋**playbook 規則外部化**（宣告式 JSON，新增規則改設定不寫程式，且不用 eval/exec）。
  - 三段式核算（申報值／依法應為／系統核算），報表明列計算式（回應「不能只說錯、要秀怎麼算」）。
  - **版本選擇器**：依報核日期自動對應 107/108/111/113 年版門檻。
- **預計研發方式**：以條文→結構化規則 spec 之半自動抽取（附法源、分類可機械檢查/半自動/需人審），生成規則一律草稿、經承辦複核＋gold 回歸測試才上線。
- **交付/查核**：逐年版 playbook ＋核算報告。

### B3　LLM Wiki RAG 法規知識庫

- **與題目之關聯**：承辦查法規耗時且易漏；法規逐年更新。RAG 讓 Agent「查得到、對得上、可回溯」。
- **技術內容**：
  - 法規知識庫（都更條例、臺北市自治條例、容積獎勵辦法，逐年版）向量化檢索；報核日期→正確年版路由。
  - 檢索結果附條文原文供協審報表引用（杜絕 AI 編造法源）。
  - 條文改版 diff → 標記「哪些規則要跟改」，維護可規模化。
- **預計研發方式**：以既有 wiki（docs/ 逐年版 Markdown）為語料建 RAG；檢索以語意＋關鍵字混合；引用一律附來源。
- **交付/查核**：RAG 法規知識庫上線（查核點 B3，Q3–Q4）。

### B4　AI 協審報表生成

- **與題目之關聯**：協審的最終產出——一份承辦看得懂、可稽核、標明「哪裡疑義、依據哪條、待覆核」的報告。
- **技術內容**：既有 HTML 報表（Jinja2）＋PDF 紅框標註（PyMuPDF）；每筆 finding 附狀態（pass/fail/warn/skip）、法源、計算式、證據頁碼；不確定項標「待覆核」。
- **預計研發方式**：整合 A/B 軌結果為單一報表；以真實案例調校可讀性與版面。
- **交付/查核**：AI 協審報表上線（查核點 B4，Q4）。

## 四、C 分項｜平台介接 ＋ 去識別化 ＋ Trust AI ＋ 場域驗證

**與題目之關聯**：讓系統真正接進都更處的實際流程（線上掛件平台），並在**個資不出境**與**可信任治理**的前提下，用真實案例證明可用。

### C1　審議服務平台資料介接

- **與題目之關聯**：北市 2025/2/10 全面線上掛件，介接平台才能讓協審進入實際作業流程、並取得結構化欄位供標註。
- **技術內容**：與臺北市都市更新審議服務平台線上申請系統之標準化資料介接（案件基本資料、辦理進度、結構化欄位）。
- **預計研發方式**：委外/協同平台系統方定義介接協議（JSON schema、欄位對應、認證）；先唯讀取得結構化欄位供 A1 回標，再雙向串接協審結果。
- **交付/查核**：平台介接完成、介接測試報告（查核點 C1，Q3）。

### C2　個資去識別化管線

- **與題目之關聯**：審議文件含地號、所有權人姓名、住址等個資；**真實案件不得出境**，必須先去識別化才進 AI 語意層。
- **技術內容**：既有 PII 偵測/遮蔽（pii_scanner/pii_masker）擴充為管線——高風險 PII 偵測→遮蔽（姓名「陳○明」式、身分證/電話後四碼、地號遮蔽）→留存對應表於地端。
- **預計研發方式**：以規則＋樣態偵測為主、VLM 輔助；遮蔽後才進語意協審；地端全程、對應表不外流。
- **交付/查核**：去識別化管線、遮蔽測試報告（查核點 C2，Q2）。

### C3　AI 可信任治理（Trust AI）機制　★ AI Agent 類別必要

- **與題目之關聯**：政府審查工具尤重可信任；申請須知明訂 AI Agent 類別須建 Trust AI 機制並於結案提檢測報告。
- **技術內容**（六面向落地）：人類監督（協審原則、人在迴路、待覆核路由）、透明可解釋（規則確定性、附法源/計算式）、隱私（地端/去識別化）、公平一致（統一年版）、安全穩健（非中資、幻覺以確定性層兜底、審計軌跡）、可課責（判定可回溯、操作日誌）。
- **預計研發方式**：建立 AI 判定信心分數 → 低信心強制人審之路由；模型/資料版本管理；異常告警與回饋重讀；結案委外第三方 Trust AI 檢測/驗證。
- **交付/查核**：Trust AI 機制 ＋ 第三方檢測報告（查核點 Trust AI，Q4，結案驗收必要）。

### C4　去識別化真實案例場域驗證

- **與題目之關聯**：以真實審議案例（去識別化）證明協審準確度與可用性，作為六都擴散之實證。
- **技術內容**：於去識別化真實案例上端到端跑協審（讀表→格式→核算→報表），比對承辦人工結果，量測逐欄準確率、格式涵蓋、時程縮短。
- **預計研發方式**：僅用公開已核定案或去識別化樣本（PII 不出境）；分批試辦【N】件；產出協審準確度驗證報告與承辦回饋。
- **交付/查核**：場域驗證報告【N】件（查核點 C4，Q4，結案驗收）。

## 五、開發方法與品質保證（貫穿全案）

- **既有系統為底、迭代開發**：以運行中系統為基線，分項 PR + 回歸測試（既有 470+ 測試），避免改壞。
- **確定性守門**：AI 生成之規則/判定一律草稿，經承辦複核＋gold 回歸測試才上線；能確定性判定者不交 AI。
- **評測驅動**：以 gold 測試集與 precision/recall harness 量化每次迭代；準確率、格式涵蓋、幻覺率均可量測。
- **主權與資安**：地端推理、資料不出境、去識別化、第三方資安＋Trust AI 檢測。
