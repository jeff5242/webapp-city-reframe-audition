系統知識架構說明

本系統採用「結構化規則引擎 + 精準 LLM 調用」的混合架構，兼具 RAG 知識管理的靈活性與 Workspace LLM 的確定性與效率。

傳統做法

RAG

Retrieval-Augmented Generation

運作方式

  1. 將法規文件切成 chunks
  2. Embedding → 向量資料庫建索引
  3. 查詢時向量搜尋 top-k chunks
  4. 將 chunks 塞入 LLM 提示生成答案

✅ 優點

  • • 知識可動態更新（重新 embedding）
  • • 能處理大量非結構化文件
  • • 不需人工整理規則

❌ 缺點

  • • 向量搜尋可能遺漏語義關聯
  • • 需向量 DB 基礎設施（Pinecone 等）
  • • 每次查詢結果可能不一致
  • • 無法保證規則完整性
  • • LLM 幻覺風險高

本系統採用

混合架構

Structured Rules + Targeted LLM

① 結構化知識庫（RAG 精神）

法規 Wiki 以 YAML 結構化儲存，版本控管、可審計、可 diff。不需向量搜尋 — 直接精確命中。

docs/wiki/111.yaml

② 規則引擎（Workspace LLM 精神）

審核邏輯以 Python 確定性規則實作，法源明確，零幻覺風險，可追溯每條判斷依據。

auditor/rules/engine.py

③ PDF 直接提取（零 LLM 依賴）

pdfplumber 直接萃取文字，Regex 解析數值。速度快、費用零、離線可用。

auditor/parsers/pdf_reader.py

④ 精準 LLM 調用（兩者結合點）

僅在規則無法確定性判斷的模糊情境下調用 Anthropic API，控制成本並最大化準確性。

anthropic SDK — 按需調用

另一種做法

Workspace LLM

Fine-tuned / System-prompted Model

運作方式

  1. 將法規知識納入系統提示或微調模型
  2. 查詢時直接推理，無需外部檢索
  3. 模型本身即「知識庫」

✅ 優點

  • • 零延遲知識存取
  • • 語境理解能力強
  • • 無需額外基礎設施

❌ 缺點

  • • 知識更新需重新訓練/調整
  • • 法規版本切換不透明
  • • 計算成本高（每次全量推理）
  • • 難以審計每條判斷依據
  • • 幻覺仍存在
